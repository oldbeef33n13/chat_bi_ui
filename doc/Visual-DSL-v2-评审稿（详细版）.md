# Visual DSL v2 评审稿（详细版）

状态：草案  
版本：v0.9  
日期：2026-03-29  
适用范围：`chart / doc / ppt` 三类可视文档协议、编辑器内核、运行时渲染链路  
评审对象：架构、前端、后端、AI、测试、存量系统负责人

---

## 1. 结论

### 1.1 总体结论

本次评审不建议继续推动“当前新工程 DSL 直接替换老 DSL”的方案。更合理的目标应调整为：

1. 老 DSL 保留为“存量业务 DSL / 外部交换协议”。
2. 新工程当前 DSL 收敛为“编辑器与运行时内核协议”。
3. 在两者之间建立长期存在的协议驱动层，而不是一次性转换脚本。
4. 最终形成新的目标协议 `Visual DSL v2`，作为后续统一建设方向。

### 1.2 面向存量系统的结论

老 DSL 不是一份“错误的协议”，而是一份偏业务结果、偏报告生成、偏渲染载荷的协议。它在以下场景依然合理：

- 报告生成结果落盘
- 存量系统之间的数据交换
- AI 生成结果承载
- 简单渲染和预览

但它并不天然适合作为“强编辑能力”的协议，因为它缺少以下关键能力：

- 稳定的节点树抽象
- 明确的布局层定义
- 内容协议与运行状态协议分层
- 强约束校验能力
- 图表编辑语义与渲染语义分离

### 1.3 面向新工程的结论

当前新工程 DSL 已经具备“编辑器内核协议”的雏形，尤其在以下方面较强：

- `VDoc / VNode` 的统一树结构
- `layout / style / data / props` 的分层建模
- 图表、表格、PPT、报告统一走节点模型
- 编辑操作走命令与 patch

但它也不应直接上升为“企业唯一公共 DSL”，原因是：

- 它对新工程前端运行时绑定过深
- 它的结构设计更偏内核，而不是跨系统交换
- 它对业务语义表达还不够完整，尤其是老 DSL 中的 `catalog / additionalInfos / lineage / requirement`

### 1.4 最终结论

推荐方案不是“老 DSL 替换新 DSL”或“新 DSL 替换老 DSL”，而是：

- 短期：保留双协议
- 中期：引入 `Visual DSL v2`
- 长期：通过 `Profile Driver` 把多协议收束到同一编辑/渲染框架

## 2. 边界

### 2.1 本次方案纳入范围

本次统一方案仅覆盖以下三类对象：

- `chart`：独立图表文档，以及嵌入在 `doc / ppt` 中的图表块
- `doc`：报告/文档型内容，目标导出为 `docx`
- `ppt`：演示文档，目标导出为 `pptx`

### 2.2 本次方案不纳入范围

以下内容不作为“内容主 DSL”的核心组成部分：

- 任务执行状态流转
- AI 推理上下文全文
- 调度与运行日志
- 导出任务状态
- 审计结果快照
- 用户会话态 UI 状态

这些内容可以存在，但应以 `runtime sidecar / annotations / audit artifacts` 的形式存在，不再与内容主协议强耦合。

### 2.3 协议职责边界

建议将协议职责明确拆为三层：

1. 内容层：文档结构、布局、样式、块定义
2. 数据层：数据源、查询、变量、过滤器、字段定义
3. 运行层：状态、附加说明、血缘、Prompt、SQL、执行信息

### 2.4 Profile 边界

`Visual DSL v2` 应明确区分三类 profile：

- `chart profile`
  - 负责图表内容定义
  - 支持绑定式建模和系列式建模
- `doc profile`
  - 负责报告/文档内容结构
  - 吸收老 DSL 的业务语义
- `ppt profile`
  - 负责演示文档绝对布局结构
  - 不反向继承老 DSL 的报告语义

### 2.5 实施边界

短期实施不要求：

- 让所有存量系统改写为 `Visual DSL v2`
- 让老 DSL 原样直接进入当前编辑器
- 一次性完成全部导出链路统一

短期实施要求：

- 给新工程一个可评审、可落地、可扩展的目标协议
- 给老 DSL 一个合理保留与演进的位置
- 给后续改造一个清晰的分阶段路径

## 3. 风险

### 3.1 现状风险

#### 风险 1：老 DSL 过于混合

表现：

- 内容、数据、状态、附加说明、血缘信息都混在同一层
- `ReportStatus / ComponentStatus` 与正文结构直接并存
- `additionalInfos / requirement` 混在章节与目录里

影响：

- 编辑器接入时边界不清
- 校验规则难以收敛
- 后续字段增加时更容易失控

#### 风险 2：老 DSL 编辑语义不足

表现：

- 图表定义偏渲染，不偏编辑
- 表格定义偏结果，不偏模型
- 组件 `layout` 与 `id` 不是强制必填
- `Section.content` 与 `Section.components` 并存

影响：

- 编辑器难以稳定处理插入、拖拽、排版和结构调整
- 任何高级编辑功能都需要额外猜测语义

#### 风险 3：当前新工程 DSL 过于内核化

表现：

- 与新工程的编辑器、运行时、命令系统绑定很深
- 许多字段是为内部运行链路设计，而不是为外部系统交换设计

影响：

- 直接推广为“统一企业 DSL”会把其他系统拖进当前工程的实现细节
- 未来出现第三类、第四类 DSL 时，内核会被迫承载外部兼容逻辑

### 3.2 迁移风险

#### 风险 4：若强推单协议替换，评审阻力会很大

原因：

- 存量系统已经使用老 DSL
- 老 DSL 已经承载外部系统协作成本
- 直接替换意味着一次性改动范围过大

#### 风险 5：若不建立中间协议层，重复适配会越来越多

原因：

- 每个外部系统都可能有自己的字段习惯和文档语义
- 如果没有 `Driver` 层，适配逻辑会散落在编辑器、运行时、导出器里

#### 风险 6：若强行让老 DSL 覆盖 PPT，会长期不稳定

原因：

- 老 DSL 没有 `deck / slide / absolute layout / master`
- PPT 的布局语义与报告类内容差别非常大

结论：

- `ppt` 不应被“老 DSL 化”
- `ppt` 必须保留独立 profile

### 3.3 设计风险

#### 风险 7：图表如果只保留一种表达，会丢掉一部分能力

只保留 `series` 的问题：

- 渲染友好，但编辑困难
- 难以做指标绑定、字段推荐、AI 自动改图

只保留 `bindings` 的问题：

- 编辑友好，但老系统接入成本高
- 对一些精细渲染场景不如 `series` 直接

结论：

- 图表必须采用“双轨模型”

## 4. 建议

本节是本次评审的核心部分。每个变更点都按“要做什么、为什么要做、怎么做、原因与取舍”展开。

### 4.1 变更点一：将“统一 DSL”调整为“统一内核 + 多 Profile”

#### 要做什么

建立新的目标协议 `Visual DSL v2`，但不要求它直接替代所有存量 DSL。  
`Visual DSL v2` 由以下三部分组成：

- 通用核心 `Common Core`
- 三个 profile：`chart / doc / ppt`
- 协议驱动层 `Profile Driver`

#### 为什么要做

因为现有两套 DSL 的职责完全不同：

- 老 DSL 更偏业务报告和生成结果
- 新 DSL 更偏编辑内核和运行时

如果继续把“统一”理解成“只保留一套字段”，最终只会得到一份越来越臃肿、越来越不可维护的协议。

#### 怎么做

1. 新工程内部保留 `VDoc/VNode` 思路，但抽象成 `VisualDoc/VNode`
2. 用 `profile` 明确区分 `chart / doc / ppt`
3. 对外只承诺 `Visual DSL v2` 的协议边界，不承诺内部实现细节
4. 老 DSL 不废弃，而是由 `doc profile driver` 接入

#### 原因与取舍

这是一个“组织成本最低、技术风险最低”的方案。  
代价是短期内不是单协议世界，但收益是：

- 不伤害存量系统
- 不污染内核
- 后续评审更容易达成一致

### 4.2 变更点二：将内容、数据、运行信息拆层

#### 要做什么

在 `Visual DSL v2` 中，把内容、数据、运行信息拆成三块：

- `root`：内容树
- `data`：数据域
- `runtime / annotations`：运行态与补充信息

#### 为什么要做

老 DSL 的主要问题不是字段多少，而是边界混杂。  
例如：

- `status` 是运行概念
- `requirement` 是生成来源概念
- `additionalInfos` 是解释和追溯概念

这些字段和正文结构不是一回事。

#### 怎么做

1. 内容树只放可编辑、可渲染的结构
2. `data` 统一承载数据源、查询、变量、过滤器
3. `runtime` 承载执行状态、节点状态等短生命周期信息
4. `annotations` 承载 Prompt、SQL、知识解释、备注等补充信息

#### 原因与取舍

拆层后会增加一点学习成本，但会显著降低：

- 字段命名混乱
- 校验复杂度
- 编辑器逻辑侵入

这是典型的“协议长期主义”设计。

### 4.3 变更点三：`doc` profile 吸收老 DSL 的业务语义

#### 要做什么

`doc profile` 不再只保留新工程当前的 `section -> block` 模型，而是明确吸收老 DSL 的以下能力：

- `cover`
- `overview / summary`
- `catalog`
- `section`
- `additionalInfos`
- `requirement`
- 字段血缘与字段显示配置

#### 为什么要做

因为老 DSL 在 `report/doc` 场景下是有明显优势的：

- 目录、章节、摘要语义天然存在
- 更符合业务人员和 AI 生成系统的表达方式
- 适合作为“报告类文档”的上层语义模型

如果完全忽略这些语义，只保留内核化的 `VNode`，会导致：

- 上层业务语义丢失
- AI 生成系统接入成本上升
- 评审中难以服众

#### 怎么做

1. `doc` 根节点使用 `document`
2. 文档结构允许 `catalog` 和 `section` 两种语义节点
3. `summary` 作为根级正式字段保留
4. `additionalInfos` 统一迁移为节点级 `annotations`
5. `requirement` 迁移到 `runtime.requirement`

#### 原因与取舍

这不是“向老 DSL 妥协”，而是把真正有业务价值的内容收编进目标协议。  
取舍点在于：

- 语义层会比当前新 DSL 更厚
- 但会让 `doc profile` 更完整、更有业务归属感

### 4.4 变更点四：明确 `catalog` 是结构节点，不是渲染块

#### 要做什么

在 `doc profile` 中引入 `kind = "catalog"` 节点，并明确它的职责是：

- 表达章节分组和目录结构
- 管理子目录和章节
- 不直接渲染为正文块

#### 为什么要做

老 DSL 里 `Catalog` 是重要的业务结构。  
如果在新 DSL 中直接删掉它，只有两种结果：

1. 老系统接入时信息丢失
2. 目录语义被强塞进 `section` 命名或额外字段中

两种结果都不好。

#### 怎么做

1. `document.children` 允许 `catalog | section`
2. `catalog.children` 允许 `catalog | section`
3. 运行时和导出器只把 `section` 视为正文页面来源
4. 大纲、目录、导航通过 `catalog` 与 `section` 联合展开

#### 原因与取舍

增加 `catalog` 会让结构看起来更复杂，但它把“目录结构”和“正文结构”分清楚了。  
这是一种正确的语义复杂度，而不是无序复杂度。

### 4.5 变更点五：图表采用 `bindings + series` 双轨模型

#### 要做什么

`chart profile` 同时支持两套定义：

- `bindings`：面向编辑器、AI、字段推荐、自动改图
- `series`：面向旧系统接入、精细渲染、显式编码

#### 为什么要做

这是本次方案里最关键的兼容点之一。

老 DSL 的图表定义优势在于：

- `series` 直观
- 渲染器容易理解
- 内联数据场景简单直接

新 DSL 的图表定义优势在于：

- 字段绑定语义更强
- 适合做智能推荐和编辑
- 更适合与数据域统一

二者不是替代关系，而是不同层次的表达。

#### 怎么做

1. `ChartSpec` 中同时保留 `bindings?` 和 `series?`
2. 约束至少出现一种
3. 新编辑器默认写 `bindings`
4. 对接老系统或需要精细渲染时可保留 `series`
5. 若两者同时存在，以 `bindings` 作为编辑语义主模型，以 `series` 作为兼容渲染提示

#### 原因与取舍

双轨模型会让图表协议稍微复杂一些，但它避免了“用一种表达硬吃全部场景”的错误。  
这是兼顾工程现实与架构纯度的必要取舍。

### 4.6 变更点六：字段定义、血缘和显示配置结构化

#### 要做什么

把老 DSL 里的以下能力结构化纳入：

- 字段定义
- 枚举值
- UI 显示配置
- 血缘来源

并禁止继续使用 JSON 字符串承载结构化字段。

#### 为什么要做

老 DSL 里的这部分信息是有价值的，但当前表达方式不理想：

- `enumValues` 是字符串
- `ui` 是字符串

这种设计短期方便，长期代价很大：

- 无法 schema 校验
- 无法编辑器直接消费
- 无法稳定 diff

#### 怎么做

1. 引入统一的 `FieldDef`
2. 引入 `FieldDisplay`
3. 引入 `FieldLineage`
4. `enumValues` 改为结构化数组
5. `ui` 改为结构化对象

#### 原因与取舍

这项变更对老系统最“有感”，因为它会影响外部写入方式。  
但它必须做，否则字段系统永远无法成为可治理资产。

### 4.7 变更点七：数据定义统一进 `data domain`

#### 要做什么

建立统一的数据域 `data`，用于承载：

- 变量
- 过滤器
- 数据源
- 查询
- 节点数据引用

并同时支持：

- 内联数据 `inlineData`
- 静态数据源 `staticData`
- 远程数据源 `remote`

#### 为什么要做

当前新 DSL 的优势在于“绑定式数据模型”，老 DSL 的优势在于“内联数据简单直接”。  
最终方案不应二选一，而应提供统一容器。

#### 怎么做

1. 文档级 `data` 作为统一数据域
2. 节点级 `NodeDataRef` 允许 `sourceId/queryId`
3. 节点级 `NodeDataRef` 也允许 `inlineData`
4. 约定优先级：
   - 节点 `inlineData`
   - `sourceId/queryId`
   - profile 默认数据策略

#### 原因与取舍

这个方案能让：

- 老系统继续直接给数据
- 新系统继续走查询与绑定
- 同一个编辑器只维护一条读取链路

### 4.8 变更点八：PPT 保持独立协议，不向老 DSL 回退

#### 要做什么

明确 `ppt profile` 使用独立结构：

- `deck`
- `slide`
- `absolute layout`
- `master`

不试图通过补字段的方式把老 DSL 变成 PPT 协议。

#### 为什么要做

因为 PPT 和报告类文档从根本上不是同一类布局模型：

- 报告是流式文档
- PPT 是页级绝对布局

强行统一只会出现大量例外规则。

#### 怎么做

1. `ppt` 根节点固定为 `deck`
2. `deck.children` 固定为 `slide`
3. `slide.children` 为绝对布局块
4. `master` 独立定义头尾、页码、强调色和留白

#### 原因与取舍

这会让 `ppt` 与老 DSL 的共享字段减少，但能保证：

- 运行时稳定
- 编辑器交互稳定
- 导出链路稳定

### 4.9 变更点九：建立 `DocumentProfileDriver`

#### 要做什么

在协议和编辑器之间增加驱动层，而不是让编辑器直接绑定某一份 DSL。

驱动层建议至少提供：

- `validate`
- `outline`
- `renderModel`
- `editOps`
- `saveback`

#### 为什么要做

如果没有这层，未来每接一种外部协议都要改：

- 编辑器
- 运行时
- 导出器
- Inspector

这会让兼容逻辑散落到全工程。

#### 怎么做

1. 定义 profile driver 接口
2. 当前新工程 DSL 实现 `native driver`
3. 老 DSL 实现 `legacy doc driver`
4. 编辑器、预览器、导出器统一通过 driver 读取能力

#### 原因与取舍

驱动层不是一次性工作，而是架构层“保险丝”。  
它的成本值得长期支付。

### 4.10 变更点十：加强版本化与校验治理

#### 要做什么

把以下能力设为目标协议的强约束：

- `schemaVersion` 必填
- 核心节点 `id` 必填
- 结构化字段禁止再用字符串 JSON
- `kind` 与 `props` 关系明确
- profile 独立 schema

#### 为什么要做

无版本、弱校验的 DSL 早期开发快，但后期治理极难。  
当前两套 DSL 都已经进入“需要治理”的阶段。

#### 怎么做

1. 所有根文档都带 `schemaVersion = "2.0.0"`
2. `profile` 作为根级识别字段
3. 给 `chart / doc / ppt` 分别建立 schema
4. 对“可接受但不推荐”的旧字段做显式标记

#### 原因与取舍

版本化会增加协议维护成本，但它是后续评审、兼容和迭代的前提。

### 4.11 变更点十一：明确不再继续扩展的旧定义

#### 要做什么

在评审里明确列出“停止扩展项”，避免以后继续模糊扩容。

建议停止扩展的点包括：

- `Partial<BasicComponent>` 作为核心组件定义
- `Section.content` 与 `Section.components` 并存
- `enumValues/ui` 用 JSON 字符串承载
- `ChartOption.eChartOption: any` 作为主定义入口
- `status` 长期混在内容主体里

#### 为什么要做

如果评审不明确把这些点冻结，团队在交付压力下会自然沿旧方式继续加字段，最终再次走向不可治理。

#### 怎么做

1. 在评审文档中明确列出“停止扩展项”
2. 在实现文档中标注“仅兼容读取，不推荐新增”
3. 新增能力一律优先进入 `Visual DSL v2`

#### 原因与取舍

这类声明看起来偏管理，但实际上是协议治理的必要手段。

### 4.12 变更点十二：采用分阶段落地策略

#### 要做什么

落地分三阶段推进：

1. 评审定版阶段
2. 内核抽象阶段
3. 双协议接入阶段

#### 为什么要做

因为本次不是单点重构，而是协议重构。  
如果不分阶段，极容易在中途重新回到“短平快补字段”。

#### 怎么做

建议阶段划分如下：

##### 阶段 A：评审定版

- 定稿 `Visual DSL v2`
- 定稿 driver 设计
- 定稿停止扩展项

##### 阶段 B：内核抽象

- 新工程内核从当前 `VDoc/VNode` 提炼到 `VisualDoc/VNode`
- 编辑器与运行时改为面向 driver

##### 阶段 C：双协议接入

- 先接 `doc`
- 再接 `chart`
- `ppt` 继续走 native profile

#### 原因与取舍

分阶段会拉长整体时间，但会显著降低重构失败概率。
