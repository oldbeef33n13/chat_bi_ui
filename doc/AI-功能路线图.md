# AI 功能路线图

## 1. 背景与目标

当前产品已经具备较完整的传统能力：

- `dashboard` 设计、预览、展示
- `report` 章节式内容编辑、预览、导出
- `ppt` 页面式编辑、预览、导出
- `templates / data-endpoints / assets / preview / export / schedules` 后端链路

下一阶段的目标，不是简单“加一个 AI 对话框”，而是构建一套真正有业务价值的智能能力：

- 帮用户更快创建内容
- 帮用户更快理解数据
- 帮用户更快形成结论和表达
- 在合适的时机给出主动建议

核心目标是：

`降低配置心智 + 提升分析深度 + 提高产出效率`

---

## 2. 总体设计原则

### 2.1 AI 的定位

AI 在本产品里优先是“副驾驶”，不是完全接管编辑器的自动代理。

优先做这 4 类能力：

- `理解意图`
  把用户的自然语言目标转成模板、图表、布局、数据接口和参数
- `理解数据`
  知道数据的结构、字段角色、适合的统计方式和展示方式
- `理解表达`
  知道 dashboard、report、ppt 各自适合怎样组织内容
- `主动辅助`
  能在数据、布局、表达层面给出有依据的建议

### 2.2 统一交互原则

AI 不应该分散成很多零碎按钮，而应收成三层：

1. `统一主入口`
   用于问、查、生成、总结、改写
2. `上下文快捷动作`
   图表旁、章节旁、页面旁的局部智能动作
3. `主动建议流`
   有依据的推荐卡片，可预览、可接受、可忽略

### 2.3 执行原则

- 所有 AI 修改必须 `可预览`
- 所有 AI 修改必须 `可确认`
- 所有 AI 修改必须 `可回退`
- AI 推荐优先输出 `结构化建议`，而不是直接改底层 DSL

### 2.4 当前阶段不优先做

- 全自动长链路 agent
- 无确认直接改整份文档
- 高风险自动执行外部动作
- 复杂多轮自治规划

---

## 3. AI 能力总览

建议把 AI 能力收成 6 大类：

1. `模板创建类`
2. `数据理解与配置类`
3. `内容表达与排版类`
4. `数据分析与总结类`
5. `主动式智能类`
6. `运行态问答与数据故事类`

---

## 4. 场景 User Case 矩阵

## 4.1 Dashboard

### 典型目标

- 快速生成大屏或 PC 工作台
- 自动补齐关键模块
- 提供即时洞察和异常解释

### User Case

| 编号 | User Case | 用户意图 | AI 输入 | AI 输出 | 是否需要确认 |
|---|---|---|---|---|---|
| D-01 | 一句话创建 dashboard | 快速建大盘 | 业务主题、时间范围、关注指标 | 页面结构、图表建议、布局草案 | 是 |
| D-02 | 从已有大盘派生区域版本 | 快速复用 | 当前模板 + 区域变量 | 新模板草案 | 是 |
| D-03 | 自动补齐缺失模块 | 页面信息不完整 | 当前 dashboard DSL | 建议新增 KPI、趋势、表格、说明块 | 是 |
| D-04 | 自动整理布局 | 页面太乱 | 当前页布局 | 排列建议、尺寸调整、空白优化 | 是 |
| D-05 | 图表结论生成 | 快速理解图表 | 图表配置 + 当前数据 | 一句话结论、异常说明 | 否 |
| D-06 | 主动风险提示 | 自动发现问题 | 当前快照与历史对比 | 风险提示卡片、建议动作 | 是 |

### 价值判断

Dashboard 最有价值的 AI 能力是：

- `快速搭结构`
- `局部补齐`
- `即时洞察`

不建议一开始就做复杂的“全页自动改版”，容易破坏用户已有布局。

---

## 4.2 Report

### 典型目标

- 快速生成结构化报告
- 帮用户把图表转成报告语言
- 优化章节顺序与叙述逻辑

### User Case

| 编号 | User Case | 用户意图 | AI 输入 | AI 输出 | 是否需要确认 |
|---|---|---|---|---|---|
| R-01 | 一句话生成报告 | 快速搭骨架 | 主题、周期、对象 | 章节结构、推荐图表、变量草案 | 是 |
| R-02 | 图表转文字 | 快速形成正文 | 当前图表和数据 | 图注、分析段落、结论 | 是 |
| R-03 | 自动写章节摘要 | 生成结论型内容 | 当前章节内容 | 摘要、关键发现、建议 | 是 |
| R-04 | 章节重组 | 优化逻辑流 | 当前章节和目录 | 新顺序、补充过渡段 | 是 |
| R-05 | 自动补分析块 | 提升分析深度 | 当前章节 + 数据 | 对比块、异常说明块、新图表建议 | 是 |
| R-06 | 生成管理层摘要 | 形成高层结论 | 整份报告 | 摘要页、行动建议 | 是 |

### 价值判断

Report 最有价值的 AI 能力是：

- `图表 -> 文本`
- `章节 -> 摘要`
- `报告 -> 管理层版本`

这是最适合形成“数据故事”的场景。

---

## 4.3 PPT

### 典型目标

- 从 report 生成汇报页
- 自动优化单页表达
- 生成讲解稿和故事线

### User Case

| 编号 | User Case | 用户意图 | AI 输入 | AI 输出 | 是否需要确认 |
|---|---|---|---|---|---|
| P-01 | 一句话生成汇报 PPT | 快速做汇报稿 | 主题、对象、时长 | slide 结构、每页标题、内容块建议 | 是 |
| P-02 | 从 report 转 PPT | 周报转汇报 | report 内容 | PPT 大纲、页面拆分、讲述顺序 | 是 |
| P-03 | 优化单页表达 | 页面太挤/太平 | 当前 slide | 重排布局、压缩文案、重点高亮 | 是 |
| P-04 | 生成演讲备注 | 辅助讲解 | 当前 slide 或 deck | speaker notes | 否 |
| P-05 | 补下一页 | 从当前页继续 | 当前上下文 | 下一页建议草案 | 是 |
| P-06 | 管理层视角改写 | 汇报对象变化 | 当前 deck | 更高层摘要和表达 | 是 |

### 价值判断

PPT 最有价值的 AI 能力是：

- `report -> ppt`
- `内容 -> 表达优化`
- `页面 -> 讲稿`

---

## 4.4 数据理解与配置

### 典型目标

- 快速理解接口
- 快速理解字段
- 快速决定怎么展示
- 快速完成字段映射与统计口径

### User Case

| 编号 | User Case | 用户意图 | AI 输入 | AI 输出 | 是否需要确认 |
|---|---|---|---|---|---|
| S-01 | 解释数据接口 | 看懂接口 | endpoint schema + sample | 参数说明、字段说明、适用场景 | 否 |
| S-02 | 推荐展示方式 | 不知道用什么图 | 字段 schema + sample rows | 趋势/对比/占比/关系图推荐 | 否 |
| S-03 | 自动字段绑定 | 减少手工配置 | 图类型 + 字段信息 | 维度/指标/系列映射草案 | 是 |
| S-04 | 自动统计口径 | 降低聚合理解成本 | 字段类型、业务语义 | 统计口径建议 | 是 |
| S-05 | 自动参数映射 | 降低变量配置成本 | endpoint params + template vars | paramBindings 草案 | 是 |
| S-06 | 数据质量提示 | 避免误配 | sample rows + schema | 缺失值、重复值、异常字段提示 | 否 |

### 价值判断

这一层是降低心智最关键的一层。  
如果数据看不懂、字段配不明白，后面所有 AI 生成能力都会不稳。

---

## 4.5 运行态问答与数据故事

### 典型目标

- 在预览/展示时快速追问
- 快速得到结论和证据
- 把图表和章节串成汇报故事

### User Case

| 编号 | User Case | 用户意图 | AI 输入 | AI 输出 | 是否需要确认 |
|---|---|---|---|---|---|
| Q-01 | 追问当前图表 | 理解图表 | 当前图表数据和配置 | 原因解释、异常点、建议动作 | 否 |
| Q-02 | 追问当前页/章节 | 理解内容 | 当前页或章节 | 结论/证据/建议三段式总结 | 否 |
| Q-03 | 总结整份文档 | 快速掌握整体 | 当前 snapshot | 总体摘要、风险排序、建议 | 否 |
| Q-04 | 数据故事串讲 | 形成汇报线 | 多页/多章上下文 | 背景/现状/问题/建议的故事线 | 是 |
| Q-05 | 从问答继续生成 | 从理解走向产出 | 当前对话上下文 | 新图表、新章节、新 slide 建议 | 是 |

### 价值判断

这一层最适合作为统一 AI 主入口的主要承载场景。

---

## 4.6 定时任务与主动智能

### 典型目标

- 定时执行后自动形成洞察
- 自动做周期对比
- 主动发现问题并提示

### User Case

| 编号 | User Case | 用户意图 | AI 输入 | AI 输出 | 是否需要确认 |
|---|---|---|---|---|---|
| J-01 | 定时摘要 | 跑完即有总结 | schedule snapshot | 摘要、关键指标变化 | 否 |
| J-02 | 周期对比 | 看变化趋势 | 多次 run 结果 | 本期 vs 上期变化说明 | 否 |
| J-03 | 异常告警摘要 | 自动识别异常 | 当前 run + 历史 | 异常列表、风险排序 | 否 |
| J-04 | 主动补建议 | 不只看结果 | snapshot + 模板结构 | 建议新增图表/章节/汇报页 | 是 |
| J-05 | 字段映射失效提醒 | 防止运行错误 | endpoint schema 变化 | 映射失效提示和修复建议 | 是 |

### 价值判断

主动智能适合做，但不建议一上来做得太重。  
第一阶段更适合做“有依据的主动建议”，而不是持续自治。

---

## 5. 统一 AI 入口设计

建议最终形成一套统一入口，而不是在各处堆独立 AI 按钮。

## 5.1 三层交互模型

### 第一层：统一主入口

一个统一的 AI 助手入口，支持四类动作：

- `问`
- `查`
- `生成`
- `改`

### 第二层：上下文快捷动作

按对象提供贴近场景的智能动作：

- 图表旁：
  - `解释图表`
  - `生成标题`
  - `生成一句结论`
  - `推荐更适合的图`
- 章节旁：
  - `补总结`
  - `补异常分析`
  - `重排章节`
- 页面旁：
  - `优化布局`
  - `改成管理层风格`
  - `补下一页`

### 第三层：主动建议流

非打扰式推荐卡片，需满足：

- 有明确依据
- 可预览
- 可接受
- 可忽略

---

## 6. 数据故事交互设计

“数据故事”建议做成一条独立能力线，不只是普通问答。

## 6.1 三层故事表达

### 层 1：一句话洞察

例如：

- `本周华东区域重大告警较上周上升 18%，集中出现在 3 月 4 日晚间窗口。`

### 层 2：三段式总结

- `结论`
- `证据`
- `建议`

### 层 3：汇报故事线

- `背景`
- `现状`
- `问题`
- `原因`
- `建议`

## 6.2 典型 User Case

- `把这张图转成一句话洞察`
- `把这一章转成三段式摘要`
- `把整份 report 转成领导汇报故事线`
- `给当前 PPT 页生成讲稿`

---

## 7. AI 功能路线图

建议分 `P0 / P1 / P2` 三个阶段推进。

## 7.1 P0：高价值、低风险、强落地

### 目标

先把“理解数据 + 快速生成 + 快速总结”做起来。

### 功能

- 一句话创建 `dashboard / report / ppt`
- 数据接口说明
- 字段角色识别
- 推荐展示方式
- 自动字段绑定草案
- 图表结论生成
- 章节摘要生成
- 局部布局优化

### 对应入口

- 编辑器统一 AI 主入口
- 图表数据面板中的推荐卡片
- 图表/章节/页面旁的上下文动作

### 说明

这是最值得尽快落地的一期。  
它能明显降低使用门槛，又不会因为自动化过重而失控。

---

## 7.2 P1：跨文档表达与业务闭环增强

### 目标

从“辅助配置”升级到“辅助表达”和“辅助产出”。

### 功能

- report -> ppt 自动转写
- report 管理层摘要
- PPT 讲稿生成
- 定时任务自动摘要
- 周期对比总结
- 主动异常发现
- 页面/章节/slide 重排建议

### 对应入口

- 报告详情页
- PPT 编辑页
- 定时任务运行历史页
- 统一 AI 入口中的“生成”和“总结”

### 说明

这一期开始形成真正的“数据故事”能力。

---

## 7.3 P2：主动智能与持续助手

### 目标

让 AI 从响应式工具升级成更主动的助手。

### 功能

- 主动发现数据异常
- 主动提示模板结构缺失
- 主动建议补图、补章节、补 slide
- 交互式故事线构建
- 多轮追问与持续上下文理解
- `nl2sql_rest -> data-endpoint` 智能生成闭环

### 对应入口

- 统一 AI 入口
- 定时任务结果页
- 展示态问答入口
- 数据接口管理页

### 说明

这一期风险更高，必须建立在 P0/P1 已经稳定的基础上。

---

## 8. 前端入口建议

不同 AI 能力应放在不同位置，不建议全堆在聊天框里。

## 8.1 编辑器统一入口

建议放在三类编辑器统一顶部：

- `AI 助手`

入口支持：

- 问问题
- 生成内容
- 总结当前对象
- 优化当前布局

## 8.2 数据属性面板

适合承载：

- 数据接口说明
- 推荐展示方式
- 自动字段绑定
- 统计口径建议

## 8.3 上下文快捷动作

适合承载：

- 图表结论
- 补图注
- 补摘要
- 优化布局
- 转管理层表达

## 8.4 展示态入口

适合承载：

- 问当前图
- 问当前页
- 问整份文档
- 生成讲稿或汇报摘要

---

## 9. 后端能力建议

要支撑这些 AI 功能，后端建议逐步补齐以下能力：

### 9.1 统一 AI 请求对象

建议统一输入结构：

- 当前模板/文档快照
- 当前选中对象
- 当前数据接口 schema
- 当前样例数据
- 当前运行变量
- 用户自然语言意图

### 9.2 统一 AI 输出结构

不要直接返回自由文本，应支持结构化输出：

- `explanation`
- `insight`
- `chartRecommendation`
- `fieldBindingSuggestion`
- `layoutSuggestion`
- `templateGenerationPlan`
- `docPatchProposal`

### 9.3 审计与回放

AI 生成和建议应保留审计：

- 输入上下文摘要
- 模型输出
- 用户是否接受
- 最终执行动作

这会是后续优化 prompt 和策略的重要基础。

---

## 10. 风险与边界

### 10.1 主要风险

- 数据理解错误，导致推荐错误
- 自动字段绑定不稳定
- 主动建议过多，打扰用户
- 直接改文档造成不可控结果

### 10.2 边界建议

- AI 不直接覆盖整份文档
- 默认只作用于当前对象、当前页或当前章节
- 全局修改必须二次确认
- 生成内容默认先预览，再应用

---

## 11. 成功指标

建议从以下指标评估 AI 是否真正有价值：

### 效率指标

- 从新建到完成首版 dashboard/report/ppt 的平均时间
- 单个图表从选数据到完成配置的平均时间
- 图表字段绑定的平均操作步数

### 质量指标

- 推荐展示方式被采纳的比例
- 自动字段绑定被保留的比例
- 自动摘要被直接采用的比例

### 体验指标

- 用户发起 AI 请求的频次
- AI 建议被接受的比例
- AI 修改被撤销的比例

---

## 12. 推荐实施顺序

建议下一阶段按以下顺序推进：

1. `P0-A`
   数据说明 + 推荐展示方式 + 字段绑定建议
2. `P0-B`
   图表结论生成 + 章节摘要生成
3. `P0-C`
   一句话生成 dashboard/report/ppt 草案
4. `P1-A`
   report -> ppt 转写
5. `P1-B`
   定时任务自动摘要与异常说明
6. `P2`
   主动式建议与交互式数据故事

---

## 13. 当前结论

对当前产品最合适的 AI 路线，不是先做重型 agent，而是先做好三件事：

- `让用户看懂数据`
- `让用户更快生成内容`
- `让用户更快形成结论`

只要这三件事打稳，后面的主动智能、数据故事、AI 协作编辑才会真正有价值。
