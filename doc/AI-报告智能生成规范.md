# AI 报告智能生成规范

## 1. 目标

报告智能生成的目标，不是直接一口气生成整篇报告正文，而是：

- 先生成合理的报告结构
- 再逐章生成内容
- 最后再生成摘要和润色内容

核心原则：

`先目录，后章节计划，再章节内容，最后摘要与润色`

---

## 2. 使用场景

适用场景：

- 一句话生成周报/月报/专题报告
- 从数据接口自动组织章节
- 从已有 dashboard 生成报告
- 生成管理层摘要版
- 对已有章节做智能补写

---

## 3. 生成阶段

## 3.1 阶段一：报告骨架生成

目标：

- 明确整份报告的逻辑结构

输出内容：

- 报告标题
- 报告目标
- 受众
- 时间范围
- 章节目录
- 每章一句话目标

示例：

```json
{
  "title": "网络运行周报",
  "audience": "ops_manager",
  "sections": [
    { "title": "总体结论", "goal": "先给领导一句话结论" },
    { "title": "告警分析", "goal": "说明告警趋势和异常点" },
    { "title": "容量分析", "goal": "说明链路负载和风险" },
    { "title": "工单与变更", "goal": "说明处置效率和变更影响" }
  ]
}
```

## 3.2 阶段二：章节计划生成

目标：

- 为每章生成内容蓝图

每章应先生成：

- 本章回答什么问题
- 使用哪些数据接口
- 推荐哪些块
- 块的顺序

块类型建议：

- 章节标题
- 摘要块
- 图表块
- 表格块
- 结论块
- 建议块

示例：

```json
{
  "sectionTitle": "告警分析",
  "question": "本周告警是否有异常上升，主要集中在哪些区域和级别",
  "blocks": [
    { "type": "summary" },
    { "type": "chart", "goal": "看告警趋势" },
    { "type": "chart", "goal": "看区域对比" },
    { "type": "table", "goal": "列出异常明细" },
    { "type": "conclusion" }
  ]
}
```

## 3.3 阶段三：章节内容生成

目标：

- 逐章生成实际内容

本阶段可生成：

- 图表
- 图注
- 分析段落
- 章节摘要
- 结论建议

要求：

- 每章应独立生成、独立预览、独立重做
- 不建议直接一次性生成全篇正文

## 3.4 阶段四：全局摘要与润色

目标：

- 在已有章节内容基础上生成：

- 封面摘要
- 管理层摘要
- 执行建议
- 章节间过渡文案

---

## 4. AI 可控制范围

AI 可以直接控制：

- 报告目录结构
- 章节标题与目标
- 章节块类型组合
- 图表与表格建议
- 分析文字
- 摘要和结论

AI 不建议直接控制：

- 整份报告所有细节样式
- 任意低层布局参数
- 用户已有章节的不可逆全局重写

---

## 5. 用户确认点

报告生成至少要有三个确认点：

## 5.1 确认点一：报告目录

用户确认：

- 章节是否合理
- 是否需要增删章节

## 5.2 确认点二：章节计划

用户确认：

- 每章要放哪些块
- 每章的数据分析重点是否正确

## 5.3 确认点三：摘要与结论

用户确认：

- 关键结论是否准确
- 建议是否符合业务语气

---

## 6. 报告智能生成 API 建议

建议拆成 4 类能力：

## 6.1 `report_outline_generate`

输出：

- 报告骨架
- 章节目录

## 6.2 `report_section_plan`

输出：

- 单章内容蓝图
- 块组合

## 6.3 `report_section_generate`

输出：

- 单章图表、文字、表格内容

## 6.4 `report_summary_generate`

输出：

- 全局摘要
- 管理层版摘要

---

## 7. 当前结论

报告生成必须分阶段。  
最重要的一点就是：

`先生成章节，再生成内容`

这样才能做到：

- 可控
- 可预览
- 可局部修改
- 可回退
