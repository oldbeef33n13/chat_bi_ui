# AI-Copilot交互设计规范

## 1. 文档目标

本文档用于固定 ChatBI 的 Copilot 交互形态，明确以下边界：

- 传统业务能力必须独立运行，不能被 AI 入口绑死。
- AI 能力必须以独立的 Copilot Shell 承载，而不是散落在多个业务面板中。
- Copilot 必须随场景变化感知上下文，并输出可操作的结构化成果，而不只是聊天文本。
- 后续实现、评审、验收，均以本文档为准。

本文档聚焦交互形态与产品规格，不展开后端编排与模型细节。

## 2. 设计结论

以下结论视为当前阶段已定方案：

1. AI 不再以“固定侧栏页签”为最终形态，而以全局浮动 `Copilot Shell` 为最终形态。
2. 传统编辑、运行、发布、导出、调度能力必须在 Copilot 关闭时完整可用。
3. Copilot 是独立协作层，不侵入主业务布局，不影响现有传统工具栏、属性面板、运行页结构。
4. Copilot 必须是场景感知的，不同场景暴露不同能力集。
5. Copilot 必须显示上下文，不允许完全隐式传递。
6. Copilot 主交互必须收敛成 `简洁上下文 + 单线程对话 + 底部输入框/快捷动作` 的低心智模型。
7. Copilot 必须支持“跟随选择”和“锁定上下文”两种模式。
8. 聊天消息是主视角，结构化结果是二级结果区，不允许与主对话并列成复杂操作面。
9. Copilot 输出仍必须结构化，优先输出 `proposal / artifact / insight / outline` 等结果卡片。
10. 每一轮对话默认只返回 `一句解释 + 一张主操作卡`，避免同轮出现多块竞争操作区。
11. AI 能力必须有所为有所不为；超出当前场景支持范围时，必须明确提示“我还在成长中”，并推荐可支持的动作。

## 3. 逻辑分析

### 3.1 为什么不能继续沿用固定 AI 面板

固定 AI 面板的问题在于：

- 它与编辑器布局强耦合，无法覆盖运行态、预览态、文档中心等其他场景。
- 用户一旦关闭或隐藏面板，很容易把 AI 能力理解成某个局部工具，而不是全局协作助手。
- 编辑态和运行态所需能力完全不同，固定面板会把不同场景揉成一套混杂交互。

结论：

- 固定 AI 面板可以作为过渡实现保留一段时间。
- 但最终必须收敛为全局浮动 Copilot Shell。

### 3.2 为什么 Copilot 必须独立于传统能力

如果 AI 和主场景深度耦合，会出现以下问题：

- AI 故障会污染主业务链路。
- 用户关闭 AI 后，布局或操作路径发生变化。
- 传统业务验收与 AI 验收耦合，风险扩大。

因此必须坚持：

- 主场景是主系统。
- Copilot 是外挂协作层。
- Copilot 可关闭、可最小化、可禁用。
- 主链路不能依赖 Copilot 才能完成基本编辑和运行。

### 3.3 为什么必须显式展示上下文

如果上下文完全由系统隐式传递，用户会持续出现两个问题：

- 不知道 AI 正在看哪个对象。
- 不知道为什么一句话改错了对象。

因此 Copilot 必须显式展示：

- 当前场景
- 当前文档
- 当前对象
- 当前变量/过滤条件
- 是否跟随选择

### 3.4 为什么需要“跟随选择 / 锁定上下文”

用户在编辑态和运行态会频繁点击不同对象。

如果上下文永远自动跟随：

- 正在进行中的对话会被主界面选中行为打断。

如果上下文永远锁定：

- Copilot 会失去“选中即聚焦”的高效体验。

因此必须同时支持：

- `跟随选择`
- `锁定上下文`

### 3.5 为什么主交互必须进一步收敛

如果把上下文、能力条、结果区、动作区同时摊开，会出现新的问题：

- 用户第一次打开时不知道先点哪里。
- 同一能力在多个区域重复出现，理解成本过高。
- AI 看起来像复杂工具箱，而不是会话式协作入口。

因此当前阶段必须收敛为：

- 顶部只保留一条简洁上下文
- 中间只保留对话消息流
- 底部只保留一个输入框和少量快捷动作
- 结构化结果以内嵌操作卡承接，不再单独占一个“最近结果”区
- Header 已经显示场景标签时，上下文摘要不再重复场景名，只保留对象、变量摘要和跟随状态

### 3.6 为什么结果必须结构化

纯文本聊天有两个上限：

- 不能直接预览、应用、拖拽、插入。
- 不能自然回写到文档和运行态。

因此 Copilot 输出必须优先落为：

- `Patch Proposal`
- `Artifact Card`
- `Insight Card`
- `Outline Card`

聊天只是解释层，结构化结果才是执行层。

### 3.7 为什么要收敛成“文本 + 单张操作卡”

如果一轮回复里同时出现：

- 长段解释
- 多个按钮区
- 结果区跳转提示
- 独立能力面板

用户会重新陷入“先理解系统，再完成操作”的高心智模式。

因此当前阶段统一约束为：

- Copilot 先用一句话说明自己准备做什么
- 然后只给一张主操作卡
- 这张卡直接承载预览、进入编辑、插入、更新、拖拽等主动作

典型卡片类型包括：

- `Template Search Card`
- `Patch Preview Card`
- `Generated Artifact Card`
- `Insight Card`

## 4. 总体交互架构

### 4.1 双层结构

系统分为两层：

#### 主业务场景层

- Dashboard/Report/PPT 编辑
- Runtime 运行态
- 发布
- 导出
- 调度
- 数据接口管理

#### Copilot 协作层

- 浮动窗口
- 场景感知
- 对话与结果卡片
- 内部结果状态
- 结构化结果应用

### 4.2 Copilot Shell 定位

Copilot Shell 是一个全局浮动聊天工作台，而不是复杂的多区块控制台。

它承担：

- 用简洁方式显示当前场景和上下文
- 承载输入与对话流
- 提供必要但收敛的结构化动作
- 管理最小化、拖动、缩放、停靠
- 对话主文案、确认提示、按钮文案应优先由 AI 服务结构化下发，前端只负责最简单展示与兜底

## 5. Shell 规格

### 5.1 基本形态

Copilot Shell 为全局浮动窗口，默认停靠在右下区域。

必须支持：

- 打开
- 关闭
- 最小化
- 恢复
- 拖动
- 缩放
- 位置与尺寸记忆

### 5.2 Launcher

右下角必须保留独立 AI launcher。

Launcher 规格：

- 默认始终可见
- 点击展开 Copilot Shell
- 最小化后回到 launcher 态
- 可显示当前场景色标
- 可显示未处理结果数量

### 5.3 Shell 结构

Shell 内部分为四个区块：

#### A. Header

展示：

- `Copilot` 标题
- 当前场景标签
- 当前聚焦对象/文档的一句话提示
- 跟随选择开关
- 最小化按钮
- 关闭按钮

#### B. Context Summary

展示一条精简上下文：

- 当前场景
- 当前聚焦对象或文档
- 是否跟随选择
- 必要时补充变量或线程摘要

#### C. Conversation Body

展示：

- 用户消息
- Copilot 消息
- 单张主操作卡
- 必要时的任务中间态

约束：

- 同一轮消息最多只展示一张主操作卡
- 不允许把多个结构化结果并列塞进同一轮回复
- 结果状态只作为内部承接，不再抢占主交互入口

#### D. Composer

包含：

- 输入框
- 快捷动作

约束：

- 只允许少量快捷动作
- 快捷动作默认一键发起，不再要求“点快捷动作后再点发送”
- 不允许再叠加大块能力条、复杂模式切换和多区域表单
- 对最终会改文档的操作，确认仍必须发生在消息里的操作卡上

### 5.4 推荐布局

建议布局如下：

```text
+--------------------------------------------------+
| Copilot | Dashboard编辑 | 跟随 | _ | x         |
+--------------------------------------------------+
| 正在围绕: 告警趋势图 · Dashboard编辑 · 跟随选择  |
+--------------------------------------------------+
| 对话流 / 用户消息 / Copilot 消息                  |
|                                                  |
| “我先按当前图表做了一版修改预览。”               |
| [修改预览卡：图表预览 + 更新按钮]                 |
|                                                  |
| 输入框............................ [发送]         |
+--------------------------------------------------+
```

## 6. 主回复规格

### 6.1 标准回复结构

每轮 Copilot 回复固定为：

1. 一句自然语言说明
2. 一张主操作卡
3. 必要时的 1 到 2 条简短补充

不允许：

- 同时弹出多个大型结果面板
- 把动作分散在聊天区和独立工具区
- 让用户先理解 proposal / plan / artifact 等内部术语再操作

### 6.2 主操作卡类型

#### Template Search Card

适用场景：

- 文档中心
- 用户说“我要编辑 xx 模板”

卡片内容：

- 匹配模板列表
- 模板名称
- 模板类型
- 更新时间
- `进入编辑`

#### Patch Preview Card

适用场景：

- 已选中对象
- 用户说“改标题 / 改成柱状图 / 加一个 x 维度 / 汇总成占比图”

卡片内容：

- 修改后预览
- 一句变更摘要
- `更新`
- `取消`

交互要求：

- 点击 `更新` 后，左侧主场景立即更新
- 更新完成后，Copilot 给出一句轻量确认

#### Generated Artifact Card

适用场景：

- 用户说“帮我查询 xx 数据”
- AI 基于当前文档数据源生成候选图

卡片内容：

- 候选图预览
- 一句生成说明
- `插入到当前位置`
- 在支持的场景下允许拖拽到主场景

### 6.3 不支持场景的处理

当用户请求超出当前范围时，例如：

- 发布
- 导出
- 调度
- 邮件通知
- 跨文档复杂编排

Copilot 必须：

- 明确提示“我还在成长中”
- 解释当前不支持的原因
- 推荐当前场景里可支持的 2 到 3 个动作

不允许：

- 模糊应答
- 假装支持
- 返回不可执行的复杂步骤让用户自行猜测

## 6. 场景模型

Copilot 必须随场景变化展示不同能力与默认上下文。

### 6.1 Dashboard 编辑态

默认支持：

- 修改当前图表
- 生成新图表
- 生成文本卡片
- 调整布局
- 批量改风格

默认上下文：

- 当前 dashboard
- 当前选中节点
- 当前筛选条件

结果类型：

- Patch Proposal
- Chart Artifact
- Text Artifact

### 6.2 Report 编辑态

默认支持：

- 一句话生成大纲
- 生成章节
- 重写章节
- 插入图表/表格/文本
- 调整故事线

默认上下文：

- 当前 report
- 当前章节
- 当前选中块

结果类型：

- Outline Card
- Section Artifact
- Patch Proposal
- Insight Card

### 6.3 PPT 编辑态

默认支持：

- 生成页纲
- 生成下一页
- 重写当前页表达
- 管理层表达优化
- 插入图表/结论页

默认上下文：

- 当前 deck
- 当前 slide
- 当前选中元素

结果类型：

- Outline Card
- Slide Artifact
- Patch Proposal

### 6.4 Runtime 运行态

默认支持：

- 当前图解释
- 当前页/章节总结
- 整份文档总结
- 下钻分析
- 结果转为图表/章节/slide

默认上下文：

- 当前运行文档
- 当前 scope
- 当前变量
- 当前 runtime filter

结果类型：

- Insight Card
- Analysis Card
- Chart Artifact
- Section/Slide Artifact

## 7. 上下文规格

### 7.1 上下文层次

Copilot 上下文分为四层：

#### Route Context

- 当前页面
- 当前模式
- 当前文档类型

#### Document Context

- docId
- revision
- template variables
- runtime variables

#### Selection Context

- 当前对象
- 所属章节/slide
- 选中范围

#### Conversation Context

- 当前 thread
- 最近命中对象
- 当前任务
- 最近一次可复用结果

### 7.2 跟随选择与锁定

Copilot 必须支持两种模式：

#### 跟随选择

- 主场景选中对象变化时，Copilot 上下文自动更新
- 适合快速编辑与快速分析

#### 锁定上下文

- 当前线程持续围绕当前对象
- 主界面再选中其他对象，不自动切换

默认策略：

- 首次打开 Copilot 时默认 `跟随选择`
- 用户手动切换到 `锁定上下文` 后，线程内保持锁定，直到用户主动恢复

### 7.3 Context Chips

Context Bar 至少包含：

- `对象`
- `章节/页面`
- `变量/过滤条件`
- `跟随当前` 或 `已锁定`

Context chip 必须可见，不允许完全隐藏在系统内部。

## 8. 结果模型

### 8.1 Patch Proposal

适用于修改已有对象。

必须支持动作：

- 预览
- 应用
- 拒绝
- 查看影响范围

### 8.2 Artifact Card

适用于新生成内容。

最小子类包括：

- Chart Artifact
- Text Artifact
- Section Artifact
- Slide Artifact

必须支持动作：

- 插入当前位置
- 替换当前对象
- 拖拽到主场景

当前第一版已落地范围：

- Dashboard artifact 可拖入 dashboard 编辑画布
- PPT slide artifact 可拖入 PPT 缩略图区并插入为新页面
- Report artifact 暂时仍以按钮式插入/替换为主，未开放自由拖入

### 8.3 Insight Card

适用于运行态总结与解释。

必须支持动作：

- 继续追问
- 转为图表
- 转为章节
- 转为 slide

### 8.4 Outline Card

适用于一句话生成后的结构确认。

必须支持动作：

- 确认大纲
- 修改某一项
- 重生成某一项
- 开始逐项生成

## 9. 成果区规格

当前阶段不再保留独立成果区 UI。

结构化结果仍需要被系统保存，但只作为内部结果状态存在，用于：

- 跨场景接力
- 编辑态回带
- 运行态转草稿
- 拖拽和插入动作的数据载体

约束：

- 默认不展示单独的“最近结果”或“成果区”面板
- 用户完成当前轮任务时，不应被迫跳到另一个结果区
- 结果默认以内嵌操作卡回到对话流
- 只有在跨场景接力时，系统才自动把对应结果重新带回当前对话

## 10. 主场景联动规格

补充原则：

- 传统能力是核心主链路，AI 只做高阶辅助，不得接管发布、导出、调度、数据接口等传统操作。
- Copilot 必须坚持低心智模型，优先提供少量明确动作，而不是开放模糊的“万能支持”。
- 对超出能力边界的请求，必须明确提示“我还在成长中”，并推荐当前场景内已支持的动作或引导用户回到传统入口。
- 这类边界控制采用双层策略：前端负责即时提示和动作收敛，Python orchestration 负责统一的权威校验与结构化 `unsupported` 返回。

### 10.1 联动原则

Copilot 不能只是“说完就结束”，必须与主场景双向联动。

### 10.2 主场景 -> Copilot

主场景应能驱动 Copilot：

- 选中图表，Copilot 自动聚焦该图
- 切换章节，Copilot 上下文更新
- Runtime scope 变化，Copilot 刷新当前分析对象

### 10.3 Copilot -> 主场景

Copilot 输出应能作用回主场景：

- Proposal 可预览并应用到当前文档
- Artifact 可插入主场景
- Insight 可转为文档内容
- Analysis 结果可继续下钻
- 已应用的 Artifact 需要显示落位信息，并支持一键定位回主场景结果
- 从对话中的操作卡执行 `应用` 或 `定位到结果` 后，主场景目标对象应触发一次短时局部 spotlight，明确告知用户变更落点

### 10.4 拖拽规则

拖拽对象不是聊天气泡，而是 `Artifact Card`。

拖拽规范：

- 每个 Artifact 必须有稳定 ID
- 拖拽时提供标准化 payload
- 主场景按目标容器消费 payload
- 目标区域在 hover 时必须明确提示落位语义，例如插入、替换、追加到后面
- 拖拽失败时不影响原始成果保留

当前第一版已落地范围：

- `dashboard` artifact 可拖入画布，作为新模块插入；hover 时会基于最近模块显示方向性预览，如“插入到某模块右侧”
- `ppt` slide artifact 可拖入左侧缩略图区的前插/后插锚点，作为新页面插入
- `report` section artifact 可拖到章节前插锚点、章节主体、章节后插锚点，分别执行前插 / 替换 / 后插
- `runtime` 主场景点击对象后会同步 Copilot 上下文，优先带入对象标签以及所属章节/页面信息
- `runtime` 最新 insight 卡片应只保留两类直接动作：`保存草稿` 和 `去编辑态继续处理`
- `去编辑态继续处理` 必须先把结果沉淀到内部结果状态，再切到编辑态并自动带回当前对话
- 结果卡在编辑态会显示 `已落在 ...` 的落位信息，并支持 `定位到结果`
- `dashboard/report/ppt` 的主落点在 hover 时都会显示明确动作提示
- `dashboard` 卡片、`ppt` 页面、`report` 章节在被应用或定位后会触发短时局部 spotlight

## 11. 典型交互流

### 11.1 Dashboard 编辑

1. 用户选中图表
2. Copilot 显示 `对象: 当前图表`
3. 用户输入“改成双轴图并补一段异常说明”
4. Copilot 返回：
   - 一个 Patch Proposal
   - 一个文本 Artifact
5. 用户预览并应用 Proposal
6. 用户将文本 Artifact 插入图表旁边

### 11.2 Report 生成

1. 用户输入一句话生成目标
2. Copilot 返回 Outline Card
3. 用户调 1 到 2 个章节标题
4. 用户点击“开始逐章生成”
5. Copilot 逐章返回 Section Artifact
6. 用户对每章执行插入、重生成、跳过

### 11.3 Runtime 分析

1. 用户在运行态选中图表
2. Copilot 聚焦当前图
3. 用户输入“为什么 3 月有尖峰”
4. Copilot 返回 Insight Card
5. 用户点击“按区域下钻”
6. Copilot 返回新图 Artifact
7. 用户选择保存草稿或回写到报告

## 15. 第一阶段验收标准

第一阶段交互验收以如下条件为准：

1. Copilot 可作为全局浮动窗打开、关闭、最小化、恢复。
2. Copilot 不影响传统编辑和运行链路。
3. Copilot 可显示当前场景与精简上下文。
4. Copilot 支持跟随选择和锁定上下文。
5. Copilot 输出至少支持：
   - Patch Proposal
   - Outline Card
   - Artifact Card 占位形态
6. Copilot 有独立 launcher。
7. 主场景关闭 Copilot 后仍可完整使用。
