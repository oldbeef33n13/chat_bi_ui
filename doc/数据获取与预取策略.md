# 数据获取与预取策略

## 目标

统一 `dashboard / report / ppt` 的首屏体验：

- 页面加载后，不需要用户再切页、切章节、滚动，系统就会自动进入首批数据加载。
- 当前界面里如果存在可取数的图表/表格，应直接显示数据或明确的 loading 态。
- AI 增量生成章节、页面、模块后，新增内容如果具备数据绑定，也应自动进入取数流程。

## 当前数据请求模型

节点级数据请求目前分三类：

1. `endpointId`
   - 前端走 `POST /api/v1/data-endpoints/{endpointId}/test`
2. `sourceId + queryId`
   - 前端通过 `DataEngine` 执行文档内数据源与查询
3. `static data`
   - 直接渲染文档内静态数据

当前请求解析入口统一收敛在：

- `src/ui/utils/node-data-request.ts`

请求状态复用收敛在：

- `src/runtime/data/data-engine.ts`
  - `execute()` 负责真正执行
  - `inspect()` 负责返回 `empty / pending / ready / error` 四态快照
  - `subscribe()` 负责按请求键订阅共享快照变化

## Fetch Eligibility

只有以下节点会进入自动取数：

- `chart`
- `table`

以下节点不会自动取数：

- `text`
- `image`
- 纯 `container / section / slide`

当前第一版仍保留一个兼容性规则：

- 如果节点没有显式 `sourceId`，但文档本身存在默认 `dataSources[0] + queries[0]`，会继续沿用现有 fallback 逻辑。
- 这可以保证旧模板和部分 AI 草稿先跑起来，但不应被视为长期的精确绑定策略。

## 两条取数链

### 1. 前端实时取数

适用于编辑态、运行态即时渲染：

- 节点挂载后，`useNodeRows` 会根据节点数据绑定触发 `DataEngine.execute`
- 同时容器会提前做一轮预取，把首批数据放入 `DataEngine` 缓存
- 如果请求已经在缓存中或已经在途，节点级加载会优先复用 `inspect()` 返回的快照
  - 有缓存：直接读缓存，不再额外打一轮 loading 抖动
  - 请求在途：复用同一执行链，不重复起请求
- 节点级 `loading / error / value` 不再完全由每个组件本地维护
  - 相同请求键的多个图表/表格会共用同一份 `DataEngine` 快照
  - 组件本地只保留对共享快照的轻量镜像和视图层转换
- 编辑态默认不启用请求防抖
  - 原因：开发态 `React.StrictMode + 懒加载编辑器` 会导致首屏请求在极短时间内被 cleanup 打断
  - 编辑态优先保证“打开即有数据”，而不是在首屏对请求做防抖
- 运行态仍可保留轻量防抖，用于吸收切页和视图抖动

### 2. 后端快照取数

适用于动态预览 / 导出：

- Java `TemplateSnapshotService` 会先执行 `endpointId`
- 将结果物化成 `dataSources / queries`
- 再把节点改写成 `sourceId / queryId`

因此在 preview/export 场景下，前端不一定会看到 `/test` 请求，因为数据可能已经在后端快照里算完。

## 首批预取策略

### Dashboard

- 进入页面后，预取当前画布全部 `chart / table`
- 原因：dashboard 是单屏主视图，用户预期是“打开即有数据”

### PPT

- 进入页面后，从当前页开始向后扫描
- 不再只看“下一页”，而是继续向后找到首个带 `chart / table` 的数据页
- 默认至少预取 `当前起点之后 2 个带数据绑定的页面`
- 这样即使第一页是封面、第二页是说明页，第三页才有图表，也会在首次进入时自动起数

### Report

- 编辑态：
  - 预取当前虚拟窗口里可见章节的 `chart / table`
  - 如果当前可见章节本身没有数据块，会继续向后扫描，直到命中首个带数据绑定的章节
  - 默认至少预取 `当前起点之后 2 个带数据绑定的章节`
- 运行态：
  - 当前实现仍会渲染整份页面
  - 但预取不再只停在“第一章 + 下一章”，而是继续向后找到首个真正带数据绑定的章节
  - 避免出现“前两章都是封面/目录/纯文本，用户滚到第三章才第一次取数”的情况

## 文档变化驱动取数

不能只依赖“页面初次加载”。

当前第一版已经将预取策略绑定到文档状态变化：

- 页面首次进入
- 切换 PPT 当前页
- Report 可见章节窗口变化
- 文档结构增量变化

这意味着：

- AI 逐章生成并插入当前文档后，新增节点如果具备真实数据绑定，会自动进入预取
- 用户手动插入新图表、新表格后，也会自动进入同一条取数链
- 文档前几页/前几章如果只是结构页、说明页，不会阻断后续首个数据页的自动取数
- 顶层应用状态不再在每次编辑时持有整份 live doc；完整文档保留在编辑 store 内，外层只维护发布/草稿所需的轻量元信息
- 编辑态草稿持久化改为延迟落盘，并在离开编辑态前补一次最终 flush，避免最后一笔修改丢失

## AI 增量生成场景

AI 生成分两类：

1. `结构草稿`
   - 只有 section / slide / block 骨架
   - 图表只是占位，不带真实绑定
   - 不应假装已经具备真实数据
2. `可运行草稿`
   - 节点已具备 `endpointId` 或 `sourceId / queryId`
   - 一旦插入主文档，就应自动进入取数

当前第一版已覆盖：

- 新增 artifact 插入到当前活跃范围后，容器级预取会重新计算
- `dashboard`：插入新模块后自动进入当前画布预取
- `ppt`：插入当前页或下一页后自动进入页级预取
- `report`：插入当前章节或下一章节后自动进入章节级预取

## 明确不做的事

第一版不做：

- 打开文档后把整份 report / ppt 所有节点全部一次性取完
- 对不具备真实绑定的 AI 草稿强行伪造完整数据语义
- 用 AI 取数替代传统 preview/export/schedule 主链路

## 后续收敛方向

- 将“fallback 到第一数据源”逐步替换成更明确的绑定状态提示
- 为 AI 生成结果补充更稳定的数据语义绑定，而不是只给占位图
- 将预取从“固定数据页/数据章数量”升级成更细的基于视口、活跃对象和真实网络成本的策略
